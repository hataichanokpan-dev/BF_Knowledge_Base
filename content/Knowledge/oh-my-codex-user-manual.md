# คู่มือใช้งาน oh-my-codex (OMX)

> [!info] คู่มือนี้เหมาะกับใคร
> เขียนสำหรับคนที่ **ติดตั้ง OMX พร้อมใช้งานแล้ว** และเคยใช้ ChatGPT หรือ Codex มาก่อน แต่รู้สึกว่าเวลาทำงานจริงมักต้องคอย approve บ่อย งานเดินไม่ต่อเนื่อง หรือไม่แน่ใจว่าควรสั่งยังไงให้มันช่วยพาโปรเจกต์ไปจนจบ
>
> **คู่มือนี้ไม่สอนการติดตั้ง** และไม่ใช่คู่มือแก้ปัญหา install/doctor ถ้า `omx` ของคุณยังไม่พร้อม ให้กลับไปดูเอกสารติดตั้งของโปรเจกต์ก่อน

---

## 1) คู่มือนี้จะช่วยคุณเรื่องอะไร

ถ้าพูดแบบสั้นที่สุด OMX คือ **"ชั้นการทำงาน"** ที่ครอบ Codex อีกทีหนึ่ง เพื่อให้คุณ:

- เริ่ม session ได้เป็นระบบกว่าเดิม
- ใช้ flow เดียวตั้งแต่ **clarify -> plan -> execute**
- เลือกโหมดทำงานให้เหมาะกับงาน
- เก็บแผน สถานะ และร่องรอยการทำงานไว้ใน `.omx/`
- ลดความมั่ว ลดการสั่งแบบกว้างเกินไป และเพิ่มโอกาสที่งานจะไปได้ไกลขึ้น

> [!tip] จำง่าย ๆ
> | ตัว | หน้าที่ |
> |-----|---------|
> | **Codex** | คนลงมือทำงาน |
> | **OMX** | วิธีจัดงาน + วิธีจัด flow + ตัวช่วยให้ session มีวินัยขึ้น |
>
> OMX **ไม่ได้มาแทน Codex** แต่ช่วยให้ Codex ทำงานแบบมีระบบมากขึ้น

---

## 2) OMX เหมาะกับใคร และไม่เหมาะกับใคร

### เหมาะกับคนที่

- ใช้ ChatGPT หรือ Codex อยู่แล้ว
- อยากให้การสั่งงานเป็นระบบขึ้น
- อยากทำงานยาก ๆ ให้เดินเป็นขั้นตอน
- ไม่อยากนั่งสั่งทีละนิดแล้วคอย approve ตลอดเวลา
- อยากรู้ว่าเมื่อไรควร "ถามให้ชัดก่อน" และเมื่อไรควร "ให้ลุยต่อได้เลย"

### อาจไม่จำเป็นสำหรับคนที่

- อยากใช้ Codex แบบตรง ๆ ไม่ต้องมี workflow layer เพิ่ม
- ทำแต่งานเล็กมาก ๆ ที่ scope ชัดอยู่แล้ว
- ไม่ต้องการระบบ plan / state / team / persistent loop

---

## 3) เส้นทางเริ่มต้นที่ควรคิดในหัว

แม้ OMX จะมีหลายคำสั่ง แต่ทางหลักจริง ๆ มีไม่กี่ตัว

### Flow หลัก

```
ถ้างานยังฟุ้ง  →  ถามให้ชัดก่อน     →  $deep-interview
ถ้างานเริ่มชัด  →  ทำแผนก่อน         →  $ralplan
พอแผนชัดแล้ว   →  เลือกโหมดลงมือ     →  $ralph หรือ $team
```

> [!quote] หัวใจของ OMX
> **Clarify -> Plan -> Execute** นี่คือหัวใจของ OMX เลย

---

## 4) เส้นทาง default ที่แนะนำ

OMX ถูกออกแบบและจูนมาหนักสุดกับ:

| เส้นทาง | สถานะ |
|---------|-------|
| **macOS / Linux** + **Codex CLI** | Default (แนะนำ) |
| Native Windows + Codex App | ยังไม่ใช่เส้นทางหลัก อาจไม่เสถียร |

> [!warning] สำหรับผู้เริ่มต้น
> ถ้าคุณกำลังเรียนรู้ OMX เป็นครั้งแรก ให้คิดก่อนว่า:
> **"ฉันกำลังเรียนเส้นทางมาตรฐานของ OMX บน Codex CLI"**
> แล้วค่อยไปสำรวจ path อื่นทีหลัง

---

## 5) คำสั่งหลักที่ต้องรู้ก่อน

> [!important] ทำไมอ่านส่วนนี้ก่อน
> คู่มือนี้ตั้งใจ **สอนคำสั่งก่อน** แล้วค่อยพาไปสู่ flow จริง

### 5.1 `$deep-interview`

| | |
|---|---|
| **ใช้เมื่อ** | เป้าหมายยังไม่ชัด / ขอบเขตยังไม่ชัด / ไม่อยากให้ agent เดาเองเยอะ / รู้สึกว่า "ถ้าลงมือเลือกมีสิทธิ์หลุดเป้า" |
| **ผลลัพธ์** | requirement ที่ชัดขึ้น ไม่ใช่โค้ดทันที |

มันจะช่วยถามกลับแบบมีโครงว่า:

- คุณต้องการอะไรจริง ๆ
- อะไรคือ in-scope / out-of-scope
- อะไรที่ agent ตัดสินใจแทนได้
- อะไรคือเกณฑ์สำเร็จ

```text
$deep-interview "ช่วยวาง requirement สำหรับระบบสมาชิกของเว็บขายของออนไลน์"
```

> [!tip] คิดแบบนี้
> `$deep-interview` ไม่ได้มีไว้ "ถ่วงเวลา" แต่มีไว้ **ลดการหลงทางตอนลงมือจริง**

---

### 5.2 `$ralplan`

| | |
|---|---|
| **ใช้เมื่อ** | requirement เริ่มชัด / อยากได้แผนที่มีเหตุผลรองรับ / อยาก review trade-off / อยากได้ PRD / test spec / acceptance criteria ที่ชัด |
| **ผลลัพธ์** | แผนที่ "พร้อมทำจริง" — scope ชัด, สิ่งที่จะทำ/ไม่ทำ, verification plan, ความเสี่ยง |
| **เปรียบเทียบ** | เปลี่ยนจาก "ไอเดีย" ไปเป็น "แผนปฏิบัติการ" |

```text
$ralplan "approve the safest implementation path for the auth module"
```

---

### 5.3 `$ralph`

| | |
|---|---|
| **ใช้เมื่อ** | มีแผนชัดแล้ว / อยากให้มี "เจ้าของคนเดียว" พางานไปจนจบ / งานต้องการ persistence สูง / อยากให้มี loop ของ execution -> verify -> fix -> verify |
| **จำง่าย ๆ** | โหมด **"เอาให้จบ"** |

เหมาะกับงานแบบ:

- เขียนเอกสารใหญ่ 1 ชุดให้เสร็จ
- แก้งานที่ต้องไล่จนผ่าน test/verification
- ทำของที่ต้องมีคนคอยถือภาพรวมต่อเนื่อง

```text
$ralph .omx/plans/prd-oh-my-codex-user-manual.md
```

> [!quote] จุดแข็งของ Ralph
> มันไม่ได้บอกว่า "พอประมาณนี้น่าจะได้" แต่จะเน้นว่า **ต้องมีหลักฐานว่าจบจริง**

---

### 5.4 `$team`

| | |
|---|---|
| **ใช้เมื่อ** | งานใหญ่พอที่ควรแตกเป็นหลาย lane / มีหลายส่วนที่ทำคู่ขนานได้ / ต้องการความเร็วจากการแบ่งงาน / ยอมรับ overhead ของการประสานงาน |

ตัวอย่างการแบ่งงาน:

| Lane | หน้าที่ |
|------|---------|
| คนที่ 1 | draft เนื้อหา |
| คนที่ 2 | เช็ก source |
| คนที่ 3 | verify acceptance criteria |

```text
$team 3:executor "execute the approved plan in parallel"
```

ถ้าเป็นงานเอกสาร ก็ยังใช้หลักคิดเดียวกัน คือแบ่ง section ownership ให้ชัด แล้วค่อยรวมงานตอนท้าย

> [!caution] จำไว้
> `$team` ไม่ได้แปลว่า "ดีกว่า `$ralph` เสมอ"
> หลายครั้งงานเอกสาร งานแก้ระบบ หรืองานที่ต้องถือภาพเดียวทั้งก้อน จะเหมาะกับ `$ralph` มากกว่า

---

## 6) คำสั่งเสริมที่ควรรู้

คำสั่งพวกนี้ไม่ใช่พระเอก แต่ช่วยมากเวลาใช้งานจริง

### 6.1 `/skills`

ใช้ดูว่า session นี้มี skill อะไรบ้าง

เหมาะเวลาคุณคิดว่า:
- "งานนี้น่าจะมี workflow สำเร็จรูปไหม"
- "มี skill สำหรับ review / security / planning ไหม"

---

### 6.2 `omx status`

ใช้ดูว่ายังมี mode ไหน active อยู่หรือไม่

เหมาะเมื่อ:
- คุณไม่แน่ใจว่ามี workflow ไหนยังค้างอยู่
- session ก่อนเหมือนยังไม่ปิด clean

---

### 6.3 `omx cancel`

ใช้ตอนอยากหยุด mode ที่กำลัง active แบบเป็นระเบียบ

เหมาะเมื่อ:
- เปลี่ยนใจ
- ต้องหยุดงานกลางคัน
- ต้องการ reset state ก่อนเริ่มใหม่

---

### 6.4 `omx explore` และ `omx sparkshell`

> [!note] ตัวช่วยรอง (สำหรับผู้เริ่มต้น)
> ถ้าคุณยังใหม่กับ OMX **อย่าเริ่มจากสองตัวนี้ก่อน**
> ให้เริ่มจากการเข้าใจ flow หลักก่อน แล้วค่อยใช้สองตัวนี้เสริม
>
> | คำสั่ง | หน้าที่ |
> |--------|---------|
> | `omx explore` | ค้น repo แบบอ่านอย่างเดียว |
> | `omx sparkshell` | inspection / bounded verification ผ่าน shell |

---

## 7) วิธีคิดก่อนสั่งงาน OMX

> [!important] หัวใจที่ทำให้ "งานไปไกล" หรือ "งานตันเร็ว"
> หลายคนคิดว่าปัญหาอยู่ที่ model ไม่เก่งพอ แต่จริง ๆ บ่อยครั้งปัญหาอยู่ที่ **คำสั่งเริ่มต้นไม่ชัด**
>
> ก่อนสั่งงาน ให้พยายามตอบ 5 ข้อนี้ให้ได้ก่อนเสมอ

### 7.1 เป้าหมายจริงคืออะไร

ไม่ใช่แค่ "อยากได้เว็บ" แต่คือ:
- เว็บนี้มีไว้ทำอะไร
- ใช้กับใคร
- ความสำเร็จหน้าตาเป็นยังไง

> [!example] เปรียบเทียบคำสั่ง
> **ไม่ดี:**
> ```text
> ช่วยทำโปรเจกต์ e-commerce ให้หน่อย
> ```
>
> **ดีกว่า:**
> ```text
> ช่วยทำระบบร้านค้าออนไลน์ขนาดเล็กสำหรับขายสินค้า 20-30 รายการ
> โดยเน้น flow ดูสินค้า ใส่ตะกร้า สรุปคำสั่งซื้อ และหลังบ้านเบื้องต้น
> ```

---

### 7.2 ขอบเขตคืออะไร

ถ้าไม่บอก scope ให้ชัด งานมักบวม

ให้ตอบว่า:
- **ต้องมีแน่ ๆ:** หน้าแสดงสินค้า, ตะกร้า, checkout mock flow
- **มีได้:** login ถ้ายังไม่ทำให้งานบาน
- **ไม่ทำแน่ ๆ:** payment gateway จริง, production deploy

---

### 7.3 Non-goals คืออะไร

> [!tip] สิ่งที่ช่วย agent มากที่สุด
> หลายครั้งสิ่งที่ช่วย agent มากที่สุดไม่ใช่ "อยากได้อะไร" แต่คือ **"ไม่เอาอะไร"**

ตัวอย่าง non-goals:
- ไม่ต้อง optimize performance ระดับ production
- ไม่ต้องทำ auth ซับซ้อน
- ไม่ต้อง deploy จริง
- ไม่ต้องรองรับหลายภาษาในรอบนี้

---

### 7.4 อะไรตัดสินใจแทนได้

ถ้าคุณอยากให้ agent เดินได้ไกลขึ้น ให้บอก **decision boundary** ด้วย

**ตัดสินใจเองได้:**
- เลือกโครงเอกสารเองได้
- เลือกตัวอย่างเองได้
- เลือกโครงสร้างโมดูลเองได้ ถ้ายังไม่หลุดจาก acceptance criteria

**ต้องถามคุณก่อนเสมอ:**
- เปลี่ยน stack ไม่ได้
- ห้ามเพิ่ม dependency ใหม่
- ห้ามแตะ production config

---

### 7.5 อะไรคือหลักฐานว่าจบ

ถ้าไม่บอกเกณฑ์สำเร็จ คุณจะได้งานที่ "ดูเหมือนจบ" แต่ยังไม่จบจริง

**ตัวอย่างเกณฑ์สำเร็จที่ดี:**
- [ ] มีไฟล์ผลลัพธ์ตาม path ที่กำหนด
- [ ] เนื้อหาครบทุก section ตาม PRD
- [ ] line count อยู่ในงบที่กำหนด
- [ ] source map ครบ
- [ ] verification check ผ่าน

---

## 8) เมื่อไรควรใช้ `$deep-interview` และเมื่อไรควรข้ามไป `$ralplan`

### ใช้ `$deep-interview` ก่อน ถ้า...

- [ ] คุณยังอธิบายโจทย์ในหัวตัวเองไม่ชัด
- [ ] ยังไม่มี acceptance criteria
- [ ] ยังไม่รู้ว่าควรให้ agent ตัดสินใจเองแค่ไหน
- [ ] งานนี้มีโอกาสหลุด scope สูง

### ข้ามไป `$ralplan` ได้เลย ถ้า...

- [x] เป้าหมายค่อนข้างชัดแล้ว
- [x] ขอบเขตค่อนข้างชัดแล้ว
- [x] รู้ว่าอะไรไม่ทำ
- [x] เหลือแค่ต้องเปลี่ยนโจทย์ให้เป็นแผนปฏิบัติการ

> [!abstract] จำง่าย ๆ
> | คำสั่ง | หน้าที่ |
> |--------|---------|
> | **Deep Interview** | ทำโจทย์ให้ชัด |
> | **Ralplan** | ทำแผนให้ชัด |

---

## 9) เมื่อไรควรเลือก `$ralph` และเมื่อไรควรเลือก `$team`

> [!note] คำถามที่คนใช้ OMX ใหม่มักสับสนที่สุด

### เลือก `$ralph` ถ้า...

- อยากให้มี owner คนเดียวถือภาพรวม
- งานยังต้องการการเชื่อมโยงต่อเนื่องสูง
- งานเขียน / งานวิเคราะห์ / งานที่ต้อง verify ทีละชั้น
- งานไม่ได้ใหญ่จนต้องแยก lane ชัด ๆ

**ตัวอย่าง:**
- เขียนคู่มือ 1 ชุดให้จบ
- ทำ PRD + doc package ให้ครบ
- แก้ feature สำคัญที่ต้อง verify หลายรอบ

### เลือก `$team` ถ้า...

- งานแตกเป็นหลายส่วนได้ชัด
- แต่ละส่วนทำคู่ขนานได้จริง
- คุณอยากได้ throughput สูงขึ้น
- ยอมรับ overhead ของการรวมงานตอนท้ายได้

**ตัวอย่าง:**
- งานเอกสารที่แบ่ง chapter ownership ได้
- งาน feature ที่ front/backend/test แยกกันชัด
- งานที่มี evidence lane, implementation lane, verification lane แยกกันได้

> [!quote] ประโยคจำง่าย
> - ถ้าอยาก **"ให้คนหนึ่งถือจนจบ"** → ใช้ `$ralph`
> - ถ้าอยาก **"ให้หลายคนทำพร้อมกัน"** → ใช้ `$team`

---

## 10) Flow หลักของ OMX ตั้งแต่เริ่มงานจนจบ

> [!important] นี่คือ flow ที่ควรฝังในหัว

### Step 1: เริ่มจากการนิยามงาน

ถามตัวเองให้ตอบได้ว่า:
- งานนี้คืออะไร
- ใครเป็นผู้ใช้
- success คืออะไร
- ไม่ทำอะไรบ้าง

### Step 2: ถ้ายังไม่ชัด ใช้ `$deep-interview`

```text
$deep-interview "ช่วยเก็บ requirement สำหรับระบบจัดการสมาชิกของร้านค้าออนไลน์"
```

ผลที่คุณอยากได้คือ requirement ที่ชัดขึ้น ไม่ใช่โค้ดทันที

### Step 3: เอาผลที่ชัดแล้วไปเข้า `$ralplan`

```text
$ralplan "approve the implementation plan for the membership system"
```

ผลที่คุณอยากได้คือ PRD / test spec / acceptance criteria ที่พร้อมใช้

### Step 4: เลือก execution mode

| ถ้า... | ใช้... |
|--------|--------|
| จะให้คนเดียวพาไปจนจบ | `$ralph` |
| จะแตกงานหลาย lane | `$team` |

### Step 5: ระหว่าง execution อย่าดูแค่ "มันทำงานอยู่"

ให้ดูว่า:
- [ ] มันยังเกาะ scope เดิมไหม
- [ ] verification plan ยังอยู่ไหม
- [ ] มันเริ่ม drift ไปทำเรื่องที่ไม่จำเป็นหรือเปล่า

### Step 6: ปิดงานด้วยหลักฐาน ไม่ใช่ความรู้สึก

งานจบที่ดีควรมี:
- [ ] ไฟล์ผลลัพธ์ครบ
- [ ] test / verification ผ่าน
- [ ] ไม่มี pending TODO สำคัญ
- [ ] reviewer/architect sign-off ผ่าน (ถ้า workflow นั้นต้องการ)

## 11) ตัวอย่างการใช้งานแบบเบา

### เป้าหมาย

อยากให้ OMX ช่วยวางโครงเอกสาร release note สำหรับ feature ใหม่

### วิธีคิด

- งานนี้ไม่ใหญ่ระดับต้องแตกหลายทีม
- แต่ยังอยากให้มี plan และ verification
- ดังนั้น flow เหมาะกับ `deep-interview -> ralplan -> ralph`

### ตัวอย่างคำสั่ง

```text
$deep-interview "ช่วยนิยาม requirement สำหรับ release note ของ feature ระบบสมาชิกใหม่"
$ralplan "turn the clarified requirement into a writing plan"
$ralph .omx/plans/prd-release-note.md
```

### สิ่งที่คุณควรคาดหวัง

1. requirement ชัดขึ้นก่อน
2. plan ชัดก่อนลงมือ
3. execution มี owner ชัด และปิดงานด้วย artifact ชัด

## 12) ตัวอย่างการใช้งานแบบโปรเจกต์ยาก

สมมติว่าคุณอยากใช้ OMX ช่วยพาโปรเจกต์ "เว็บขายของออนไลน์ขนาดเล็ก" ตั้งแต่ framing จนไปถึง prototype ที่ใช้งาน flow หลักได้

### 12.1 ถ้าสั่งไม่ดี งานจะไปแบบไหน

> [!bug] คำสั่งที่กว้างเกินไป
> ```text
> สร้าง e-commerce ให้หน่อย
> ```

ปัญหาที่มักตามมา:
- scope บวม
- agent ต้องเดาเยอะ
- ไม่ชัดว่าต้องมีอะไรบ้าง
- ไม่รู้ว่าจะถือว่างานจบตรงไหน

### 12.2 ถ้าสั่งดีขึ้น งานจะไปได้ไกลกว่าเดิม

> [!success] คำสั่งเริ่มต้นที่ดีกว่า
> ```text
> $deep-interview "ช่วยเก็บ requirement สำหรับระบบร้านค้าออนไลน์ขนาดเล็ก
> สำหรับสินค้า 20-30 รายการ โดยรอบนี้ต้องมี catalog, cart, checkout mock flow,
> admin เบื้องต้น และไม่ต้องมี payment จริง"
> ```

ต่อจากนั้น:

```text
$ralplan "approve the implementation plan and tradeoffs for the small e-commerce prototype"
```

ถ้า plan ออกมาชัดแล้ว:

```text
$ralph .omx/plans/prd-small-ecommerce.md
```

หรือถ้างานแตกได้ชัด:

```text
$team 3:executor "execute the approved small-ecommerce plan in parallel"
```

### 12.3 จุดที่ทำให้ OMX ช่วยได้มากขึ้น

คุณควรบอกตั้งแต่แรกว่า:

| ประเภท | ตัวอย่าง |
|---------|----------|
| **success** | demo flow ดูสินค้า → ใส่ตะกร้า → checkout mock ทำได้ครบ |
| **non-goals** | payment จริง, deploy จริง, auth production-ready |
| **decision boundary** | agent เลือกโครงไฟล์เองได้ แต่ห้ามเพิ่ม dependency ใหม่โดยไม่จำเป็น |

### 12.4 แล้วที่บอกว่า "ปล่อยให้มันทำทั้งคืน" หมายถึงอะไร

> [!warning] ความเข้าใจที่ถูกต้อง
> มันไม่ได้แปลว่า OMX จะ **การันตี** ว่าตื่นมาแล้วทุกอย่าง perfect เสมอ

สิ่งที่ควรเข้าใจให้ถูกคือ:

- OMX ช่วยให้คุณ set งานได้เป็นระบบกว่าเดิม
- ถ้า requirement และ plan ดี มันมีโอกาสพางานไปได้ไกลขึ้นมาก
- แต่ผลลัพธ์ยังขึ้นกับความชัดของโจทย์ ความซับซ้อนของงาน และข้อจำกัดจริงของระบบ

> [!quote] เป้าหมายที่ถูกต้อง
> ทำให้ session เดินได้นานขึ้น เดินมีวินัยขึ้น และมีโอกาสไปถึงผลลัพธ์มากขึ้น
>
> **ไม่ใช่**เชื่อว่ามันจะจบเองแบบมหัศจรรย์ทุกครั้ง

---

## 13) ความผิดพลาดที่ทำให้ OMX ทำงานไม่จบ

### 13.1 โจทย์กว้างเกินไป

> [!bug] ตัวอย่างที่ไม่ดี
> - ทำแอปให้หน่อย
> - ปรับระบบนี้ให้ดีขึ้น
> - สร้างธุรกิจออนไลน์ให้ครบ

โจทย์แบบนี้ทำให้ agent เดาเยอะมาก

### 13.2 ไม่มี non-goals

ถ้าคุณไม่บอกว่า "อะไรไม่ทำ" งานจะมีแนวโน้มบานออกไปเรื่อย ๆ

### 13.3 รีบ execute ทั้งที่ยังไม่ clarify

> [!warning] อันตราย
> หลายคนข้าม `$deep-interview` ทั้งที่โจทย์ยังไม่ชัด
> ผลคือ execution ไปเร็ว แต่ **ไปผิดทาง**

### 13.4 รีบใช้ `$team` ทั้งที่งานยังไม่พร้อมแตก

ทีมช่วยเรื่องความเร็ว แต่ถ้างานยังไม่ชัดพอ การแตกทีมเร็วเกินไปอาจทำให้:

- แต่ละ lane เข้าใจไม่ตรงกัน
- ต้องกลับมารวมงานเยอะ
- เกิด overhead โดยไม่คุ้ม

### 13.5 เอา advanced surfaces มาใช้เป็น default

เช่นหมกมุ่นกับ operator surface มากเกินไปตั้งแต่เริ่ม ทั้งที่ยังไม่เข้าใจ flow หลัก

> [!tip] สำหรับผู้เริ่มต้น ให้ยึดแกนหลักนี้ก่อน
> `$deep-interview` → `$ralplan` → `$ralph` / `$team`
>
> แล้วค่อยใช้เครื่องมือเสริมทีหลัง

### 13.6 ไม่มีเกณฑ์จบที่ชัด

ถ้าไม่กำหนดว่าอะไรคือ "done" คุณจะได้ session ที่เหมือนทำงานเยอะ แต่ไม่รู้ว่าควรหยุดตรงไหน

## 14) เช็กลิสต์ก่อนปล่อยให้ OMX ทำงานยาว ๆ

ก่อนจะกดให้มันเดินงานที่ยาวหรือซับซ้อน ให้เช็กสิ่งนี้ก่อน

> [!checklist] เช็กลิสต์ 10 ข้อ
> - [ ] เป้าหมายหลักชัดแล้ว
> - [ ] รู้ว่าผลลัพธ์สุดท้ายต้องเป็นอะไร
> - [ ] มี scope ชัด
> - [ ] มี non-goals ชัด
> - [ ] บอก decision boundary แล้ว
> - [ ] รู้ว่าจะใช้ `$deep-interview`, `$ralplan`, `$ralph` หรือ `$team` เพราะอะไร
> - [ ] มี acceptance criteria หรืออย่างน้อย success signal ที่ชัด
> - [ ] ไม่สั่งกว้างแบบ "ทำทั้งหมดให้หน่อย"
> - [ ] เข้าใจว่ามันช่วยเพิ่มโอกาสความสำเร็จ ไม่ใช่การันตีความสมบูรณ์แบบ
> - [ ] พร้อมกลับมา review evidence ตอนจบรอบ

> [!warning] ถ้าข้อไหนยังตอบไม่ได้
> ให้กลับไป clarify ก่อน

## 15) Template สำหรับสั่งงานให้ดีขึ้น

คุณสามารถใช้แม่แบบนี้ได้แทบทุกงาน

```text
เป้าหมาย:
...

ผู้ใช้/บริบท:
...

ต้องมีแน่ ๆ:
...

ไม่ทำในรอบนี้:
...

สิ่งที่ agent ตัดสินใจเองได้:
...

เกณฑ์สำเร็จ:
...
```

### ตัวอย่าง

```text
เป้าหมาย:
สร้างคู่มือภาษาไทยสำหรับผู้ใช้ OMX ที่ติดตั้งพร้อมแล้ว

ผู้ใช้/บริบท:
คนที่เคยใช้ ChatGPT หรือ Codex แต่ยังไม่เข้าใจ flow ของ OMX ดีพอ

ต้องมีแน่ ๆ:
อธิบายคำสั่งหลัก, flow หลัก, ตัวอย่างใช้งานจริง, เช็กลิสต์ก่อนปล่อยงานยาว

ไม่ทำในรอบนี้:
ไม่สอนติดตั้ง, ไม่ troubleshoot install, ไม่ทำ full command encyclopedia

สิ่งที่ agent ตัดสินใจเองได้:
เลือกโครงบท, ลำดับการอธิบาย, ตัวอย่างที่เหมาะที่สุม

เกณฑ์สำเร็จ:
มี handbook + source map ครบตาม PRD และอ่านแล้วคนทั่วไปเข้าใจ flow หลักของ OMX ได้
```

## 16) สรุปสั้นที่สุดอีกครั้ง

ถ้าคุณจำอะไรได้แค่ไม่กี่บรรทัด ให้จำสิ่งนี้:

> [!abstract] สรุป OMX
> - OMX ไม่ได้มาแทน Codex แต่มาจัด **workflow** ให้ดีขึ้น
> - ทางหลักของ OMX คือ **clarify -> plan -> execute**
> - คำสั่งหลักที่ต้องรู้คือ:
>   - `$deep-interview`
>   - `$ralplan`
>   - `$ralph`
>   - `$team`
> - ถ้างานยังไม่ชัด → อย่ารีบ execute
> - ถ้าไม่มี non-goals → งานจะบวม
> - ถ้าไม่มี success criteria → งานจะไม่รู้ว่าจบตรงไหน
> - ถ้าอยากให้มันเดินได้ไกลขึ้น → ต้องสั่งให้ชัดขึ้น

> [!quote] ประโยคที่สำคัญที่สุด
> OMX ไม่ได้ทำให้คุณ "ไม่ต้องคิด"
> แต่มันช่วยให้คุณ **"คิดเป็นระบบ แล้วสั่งงานเป็นระบบ"** มากขึ้น
>
> นั่นแหละคือสิ่งที่ทำให้โปรเจกต์ยาก ๆ มีโอกาสไปจนจบได้จริงมากขึ้น

## 17) อ่านต่ออะไรดีหลังจากจบคู่มือนี้

ถ้าคุณอ่านถึงตรงนี้แล้ว อยากไปต่อแบบใช้งานจริง ให้ลำดับแบบนี้:

| ขั้น | ทำอะไร | หมายเหตุ |
|------|---------|----------|
| 1 | ลองงานเล็ก 1 งานด้วย `deep-interview -> ralplan -> ralph` | เริ่มจากงานไม่ซับซ้อน |
| 2 | พอคล่อง flow หลักแล้ว ค่อยลองงานที่ใหญ่ขึ้น | เพิ่มขนาดทีละนิด |
| 3 | ถ้างานแตกเป็นหลาย lane ได้ชัด ค่อยลอง `team` | ใช้เมื่อพร้อมเท่านั้น |
| 4 | ใช้ `source map`, `status`, `cancel` และเครื่องมือเสริม | เมื่อคุ้นกับ flow หลักแล้ว |

> [!tip]
> ถ้าคุณทำตามลำดับนี้ คุณจะเข้าใจ OMX เร็วกว่าการกระโดดไปใช้ทุกคำสั่งพร้อมกันตั้งแต่วันแรก
